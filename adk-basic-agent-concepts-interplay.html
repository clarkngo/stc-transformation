Strategic frame

Full picture: The YouTube line is the ADK Masterclass; Example #1 is Building Your First Agent — the greeting agent that teaches Agent, Session, and Runner. STC bridge content should practice democratization thought leadership: go wider than “out-deep” developer tutorials — speak to managers, partners, and policy owners who will govern agents beside engineers — using stories and failure modes, not only terminals.

Why this example holds: It carries the philosophical questions non-developers ask (intent, instructions, accountability, wrong goals) without requiring viewers to read code. Execution risk: like CS50’s anchor instructor, STC needs one consistent on-camera owner (Dean or named faculty) for years; polish is secondary to showing up.

Deliverable shortcuts on the production page · Full strategic write-up

Example #1 is “small” in code but thick in ideas: several independent systems (Python, process environment, cloud identity, model routing, instruction state, session memory) must line up. Beginners often fixate on one line of code; most bugs are misalignment between layers. The tutorial’s Agent / Session / Runner trio is how Google names the “happy path” orchestration; the diagram below is how you debug when the path breaks.

1. Layer stack (bottom → top)

Read upward: each layer assumes the one below is already valid. Skip a layer mentally and the failure looks random. Map to Example #1: Agent ≈ configured behavior + model wiring inside the ADK runtime; Session ≈ conversation state the runtime keeps; Runner ≈ the entry path that executes turns (your run command / app harness) — still sitting on top of machine, secrets, and model policy.

Machine & process Python interpreter version, working directory (cwd), activated venv, installed packages. Interplay: wrong cwd shifts every relative path — config, imports, and “where is main?” all disagree.
enables →
Secrets & project identity .env loaded from a specific folder, GOOGLE_API_KEY, Cloud / AI Studio project, billing flags. Interplay: the key is not “magic text”; it couples your process to a remote quota + policy surface. A valid key in the wrong folder = same as missing.
enables →
ADK runtime Object that wires model id, instructions, tool hooks (later), and session plumbing. Interplay: this is what makes an “agent” differ from a script that calls generate_content once — the runtime owns repeated structure across turns.
feeds →
Model surface Model id string, region / capability constraints, token limits. Interplay: instructions and user text are compressed through this surface; change the model and the same prompt can succeed or fail for capability reasons, not “bugs.”
shapes each call →
Instruction state vs user turn Instructions = slow-moving pressure on every generation. User message = fast-moving input for this turn only. Interplay: if you debug them as one blob, you cannot tell “model drift” from “bad user example.”

Session memory (even in Basic) Light in Example #1, but the idea is already there: the runtime remembers what you do not paste back. Interplay: sessions are why multi-turn tutorials work without you manually concatenating chat history — they tie identity of the conversation to the runtime.

3. Interplay pairs (tension & contract)

Each pair is a contract: when one side changes, the other must be re-checked.

Script ↔ Agent

One-off call vs hosted behavior

A script ends when the process ends. An agent names a behavior slice the runtime can re-enter: same instructions, new user line. Confusing the two makes people “fix” prompts when the real issue is lifecycle (session, cwd, env).

Folder ↔ .env

Locality of configuration

The course repo uses per-example config. Treat that as a design choice: each lesson is a mini-deployable. A key at repo root does not automatically shadow-copy into the child folder — there is no spooky action at a distance.

Key ↔ Model id

Identity vs routing

The key answers “who is paying / allowed.” The model id answers “which cognitive engine.” A policy error on one can look like a failure on the other if you only read the last line of the stack trace.

Instructions ↔ User message

Static vs episodic input

Instructions are amortized across turns; the user message is marginal to this turn. If output is wrong, ask: is the instruction contradicting the example? Is the user message under-specified? Swap them mentally before blaming the model.

Session ↔ “It forgot”

State without spreadsheets

When learners say the model “forgot,” often they mean no session or a new session id. Basic Agent still teaches you to expect persistence as a first-class object, not a side effect of variables in your script.

Imports ↔ Tutorial copy

Version skew

Repo pins versions; your global Python might not. Interplay: pip freeze vs README is a compatibility handshake between author time and your machine time.

4. Failure cascades (why one bug looks like three)

Example cascade A — wrong cwd: .env not found → key missing → auth fails → some stacks only show “request error.” You might wrongly tweak the model id or instructions when the root cause was path locality.
Example cascade B — skipped pip install: Import error deep in ADK → assume framework broken → prompt tuning. The true graph is: environment incomplete → no runtime object → no meaningful model call at all.
Example cascade C — billing / project mismatch: Key present but rejected → generic HTTP/GRPC message. Interplay: cloud identity is not “inside” Python; it is a cross-system handshake. Fix lives in console pages, not in string edits to the user message.

5. How to teach this in one video arc

Suggested narrative arc (overlaps the production pack, but centers the diagram in the viewer’s head):

  1. Draw the stack first (even on a napkin graphic): machine → secrets → runtime → model → instructions/user → session.
  2. Annotate one happy path as a glowing line through all layers.
  3. Break one layer at a time and narrate which downstream symptoms appear — this teaches debugging as graph search, not superstition.
  4. Close by naming what Example 2 adds (tools) as new edges on the runtime node, not a unrelated topic.

Sample B-roll & on-screen frames

Flat SVG storyboard art (not real product UI). Open any card for the full vector file — safe to screen-record, drop into Keynote, or trace in Descript / Resolve.

0:00 — Hook

If you are learning Google’s Agent Development Kit from a crash course, you will see a small Python file, a model name, and a run command. That is enough to get a reply once. It is not enough to know why it broke when it breaks. In this video we are not racing through syntax. We are building one picture in your head: a stack of contracts between your laptop, your project, the runtime, the model, one user turn, and the session that remembers what happened. When you can see that stack, debugging stops feeling like superstition and starts feeling like reading a map.

0:45 — Framing

We will use the Basic Agent pattern from the community crash course and the official quickstart as our anchor examples. The code changes over time; the shape does not. Every agent framework eventually has to answer the same questions: where does configuration live, who holds credentials, what object actually calls the model, and what survives after the answer comes back. ADK answers those questions in a particular layout. Our job is to learn the layout once, then reuse it everywhere.

1:30 — The diagram (on screen: simple six-box flow)

[B-roll: draw or reveal six boxes left to right.] Imagine six stations on a conveyor belt. Station one is the machine: your virtual environment, your working directory, the Python interpreter you think you are using. Station two is secrets and project identity: environment variables, API keys, Google Cloud project IDs, anything that answers “which account and which quota bucket am I billing?” Station three is the runtime: the ADK objects that know how to assemble a request and interpret a response. Station four is the model configuration: which model ID, which safety or generation settings apply. Station five is a single turn: the system instructions plus the user message for this step. Station six is the session: the state that may carry history, metadata, or follow-up turns.

A “happy path” is a glowing line that visits every station in order. A bug is almost always “the line never reached a station,” or “the wrong value was attached at a station.” Your first debugging move is not to tweak the user prompt. It is to ask which station never got the signal.

3:30 — Machine layer

[B-roll: terminal listing python --version, which python, project folder.] The machine layer is boring on purpose. If the wrong interpreter runs, imports fail before your agent exists. If the working directory is wrong, relative paths to config or to a .env file silently point into empty air. When someone says “it works on my machine,” they are usually describing a machine layer mismatch, not a mystical AI problem. Fix the interpreter and the cwd first; it is cheap and it eliminates whole classes of false leads.

4:45 — Secrets and project layer

[B-roll: .env file with keys redacted; cloud console project selector.] The secrets layer is a contract between your process and the outside world. Missing keys do not produce thoughtful model errors; they produce transport failures or generic authentication messages. Wrong project IDs produce “access denied” in places that look unrelated to Python. This layer is not “inside” your prompt string. Treat it as infrastructure: load order matters, shell exports matter, and CI versus laptop differences matter. When debugging, verify the secret at the boundary: print whether the variable is non-empty, never print the secret itself.

6:15 — Runtime layer (ADK)

[B-roll: minimal agent file structure from the repo.] The runtime layer is where the framework earns its keep. It is the code that knows how to take your agent definition and turn it into a concrete call pattern for the model provider. If this layer is wrong, you can have perfect credentials and still never form a valid request object. Import errors here often look like “the framework is broken,” when the real story is “the environment is incomplete or the installed package set does not match the tutorial.” Read the traceback bottom-up until you find the first frame that is your entry point, then walk upward one layer at a time.

7:45 — Model configuration

[B-roll: model string in code; provider docs.] The model layer is narrower than people think. It is mostly identifiers and knobs: which model, which region or endpoint, which parameters the provider accepts. Confusion at this layer usually sounds like “wrong model for the task” or “policy blocked this content,” not like “Python cannot find module X.” If the runtime reached the model call and you get a structured error from the provider, you have already cleared the lower stations. Celebrate that; it tells you where not to waste time.

9:00 — Turn: instructions and user

[B-roll: system instruction string + user input.] The turn layer is what beginners want to edit first, because it is made of words. It matters, but it is downstream. If the lower layers failed, no instruction rewrite will create a model response. When the lower layers succeed, this layer controls tone, format, and task framing. Teach yourself to separate “formatting the task” from “proving the plumbing.” Prove plumbing with a tiny, almost silly user message once; then grow the task complexity.

10:15 — Session and memory

[B-roll: session object or log of turns.] The session layer is where continuity lives: prior turns, tool results if you add tools later, and anything the runtime attaches for the next call. Basic Agent might look stateless at first glance, but the conceptual slot is still there. If you skip understanding it, multi-turn behavior and tool loops will feel like magic. If you map it early, Example Two in the course, tools and actions, becomes “new edges that start and end on the runtime node,” not a separate universe.

11:30 — Interplay: one failure, many symptoms

Now the useful part. Break the stack deliberately in your mind. [On screen: dim boxes after the break point.] If secrets fail, the runtime may never get a healthy client. The symptom shows up as HTTP or gRPC errors that mention auth, not as bad prose from the model. If the machine layer is wrong, you may not even instantiate the runtime. If the model layer is wrong, you get provider errors after a successful handshake. If only the turn layer is wrong, you get a valid answer that misses the task. Naming which symptom family you are in tells you which station to walk back to. That is graph search on a straight line, which is the easiest graph search there is.

13:15 — Three short cascades (voice-over examples)

Cascade A — “bad answer.” Ask whether a model answer existed at all. If yes, the failure is often turn framing or evaluation, not credentials. If no answer, walk downward until you find the first hard error.

Cascade B — “import error deep in ADK.” Suspect incomplete environment or version skew before you suspect the model. Re-sync the tutorial’s dependency set on a clean virtual environment.

Cascade C — “cloud identity mismatch.” Treat project and service account as part of the secrets station. The fix lives in console pages and IAM bindings, not in string edits to the user message.

14:45 — Connect to the crash course repo

[B-roll: GitHub tree for Example 1 Basic Agent.] When you open Example One, read it through this lens. Point at each file or block and say which station it feeds. The goal is not memorizing file names; the goal is automatic classification. Within a few repos you will stop copying snippets blindly because you will feel where the contract boundaries are.

16:00 — Live demo beat (optional but strong)

[B-roll: run the agent twice: once clean, once with a deliberate mis-set env var.] First run: confirm the glowing path end-to-end with a boring user message. Second run: unset one variable or point to the wrong project, show the auth-layer symptom, then restore and show recovery. Narrate out loud which station broke. That thirty seconds teaches more than ten minutes of buzzwords.

17:30 — Bridge to tools (preview Example 2)

Tools do not replace this diagram; they add branches. A tool call is still orchestrated by the runtime, still billed against your project, still wrapped in turns, still recorded through session semantics. So when you click ahead to tool-based agents, you are extending the middle of the picture, not throwing it away. Keep the same map; draw more arrows from the runtime box outward and back.

18:45 — Close

If you take one line from this video, take this: learn ADK as a set of contracts between layers, not as a spellbook of prompts. Prompts matter after the contracts hold. Official docs and the crash course repo stay linked below; use the production checklist page if you are storyboarding your own recording. In the next video in your own learning path, pause at every error message and ask which station it belongs to before you change a single word of instruction. That habit will save you hours.
