City University of Seattle — Courses, Programs, Syllabi, Faculty.

Convert open-learning engagement into formal enrollment, micro-credentials, or degree pursuit.

Touchpoint Implementation Matrix

Six nodes — integration strategy, touchpoints, and the technical hook behind each
🎥

YouTube

Video Lectures & Concept Explainers
Strategy
Inbound discovery & top-of-funnel concept validation.
  • Video description CTA hooks

    Dynamic URL redirects pointing directly to matching GitHub starter repos or Open edX micro-courses.

  • Pinned comments & QR codes

    Embedded in slides to capture audience interest during specific timestamps.

  • Interactive cards / end screens

    Directing viewers to the corresponding Open edX pilot or GitHub hands-on lab environment.

Technical integration YouTube Data API — syncing video metadata/playlists directly to the Open edX course units or course web pages.
Data & KPIs available
  • Age / gender bands
  • Geography
  • Device type
  • Traffic source
  • Audience retention
  • End-screen / card CTR

All aggregate, never individual — YouTube exposes no per-viewer identity by design. See the Identity & tracking layer breakdown in Next Steps for the hard ceiling this creates.

🐙

GitHub

Lab Environments & Starter Code
Strategy
Practical application & technical asset distribution.
  • Template repositories

    One-click "Use this template" / GitHub Codespaces launcher buttons embedded directly in Open edX units and YouTube video descriptions.

  • Automated feedback

    GitHub Actions running automated test suites on student submissions to provide instant code grading before official LMS submission.

  • Static redirection pages

    Decoupled GitHub Pages routing layer to manage stable lab URLs across varying term setups.

Technical integration GitHub Webhooks & OAuth / LTI links into Open edX / LMS to track student progress and lab completions.
Data & KPIs available
  • Repo clones / forks
  • "Use this template" clicks
  • Codespace launches
  • Commit activity
  • Test-suite pass rate
  • Stars / watchers

Requires Actions/Webhooks actually instrumented per repo — not automatic on a plain template repo.

🎓

Open edX

Course Pilots & Completion Analytics
Strategy
Asynchronous self-paced validation & pilot testing ground.
  • Modular XBlocks / LTI tools

    Embed YouTube video embeds, embedded GitHub Codespaces/Gists, and live interactive widgets within course units.

  • Completion signals

    Automated completion certificates triggering an automated email nurture sequence towards full degree enrollment or micro-credential registration.

Certificate copy
Set per-course in Studio → Settings → Certificates → Certificate Web/HTML View Overview. No template work needed — this is a text field:
This certifies that {{accomplishment_copy_name}} has successfully completed {{accomplishment_copy_course_name}}, a micro-credential offered through CityUX — City University of Seattle's open micro-credentialing and workshop ecosystem.
Branding note
Certificate text can reference CityUX now. Hold off on putting a CityUX logo on the certificate itself until the visual identity is settled — a certificate feels permanent to the person holding it, unlike a Discord server name (see the pilot-server note in Community Hub).
Technical integration Open edX Analytics Pipeline / xAPI data streams feeding learner progression data into the university CRM/LMS.
Data & KPIs available
  • Enrollment counts
  • Module completion rate
  • Quiz / assessment scores
  • Time-on-task
  • Certificate issuance
  • Full xAPI event stream

The richest and most identity-linked data source of the six nodes — this is why it's the recommended profile hub in Next Steps.

Data & KPIs available
  • Registration count
  • Attendance (Zoom/Teams logs)
  • Exit-ticket responses
  • CRM lead records

Community Hub

Demand Signals & Pain Points
Pilot server
A pilot Discord is already running as proof of concept — discord.gg/PFQJNpS9A. Treat this as a working test, not the permanent home: server name, structure, and invite link are all expected to change before public launch.
Strategy
Continuous feedback, community-led discovery, & direct student support.
  • Discussion channels

    Discord (pilot running — discord.gg/PFQJNpS9A), Slack, or GitHub Discussions — dedicated spaces categorized by topic, lab help, and career advice.

  • Feature request / topic upvoting

    Upvote boards where community members post pain points or request specific technical deep dives.

Channel structure
Starting channel set for the pilot Discord, built on the Study Group template:
  • #announcements

    Course, workshop, and seminar updates — read-only, faculty/staff posting.

  • #general

    Open discussion and introductions.

  • #lab-help-[topic]

    One per active module/track — mirrors the "lab help" touchpoint, keeps GitHub/Open edX questions searchable by subject.

  • #career-advice

    Industry Q&A, resume and job-pathway discussion — where alumni and practitioners give back.

  • #showcase

    Learners post what they built — the strongest BOFU engagement signal to watch on the analytics dashboard.

  • #suggest-a-topic

    Feature-request / pain-point upvoting via 👍 reactions to start; a bot (e.g. Upvote) can formalize this later.

Moderation & guidelines
Governance policy: moderate on conduct, not sentiment. Negative feedback is a designed signal (see #suggest-a-topic), not something to remove — removing legitimate criticism is a bigger brand risk than the criticism itself.
Pinned disclaimer — post to #general and #announcements Before you post — what this server is (and isn't). CityUX Community is a peer space, not an official CityU support channel. Anything affecting your official record — grades, credit, enrollment status, accommodations, or account/access issues — needs to go through official channels ([course support / advisor contact — confirm before publishing]), not this server. What this space is for: peer help, career talk, sharing what you're building, and real feedback — #suggest-a-topic exists specifically so pain points get heard.
  • Be respectful

    No harassment, hate speech, or personal attacks — on staff, faculty, or fellow learners.

  • No brigading or pile-ons

    Directing group backlash at a specific person isn't welcome, even in response to a bad experience.

  • No doxxing or private info

    Don't share personal information about yourself or others — including screenshots of private DMs or official correspondence.

  • No spam or off-topic self-promo

    Keep promotion to designated channels; unrelated advertising gets removed on sight.

  • Criticism is welcome — make it specific

    Negative feedback about the course, platform, or program stays up as long as it's about the thing, not a person, and follows the rules above. Specific beats vague — it's more likely to get acted on.

  • Official disputes go to official channels

    Grade disputes, credit questions, and enrollment issues aren't resolved here — see the disclaimer above.

  • Enforcement is a ladder, not a hammer

    Warning → temporary mute → removal, scaled to severity and repeat offenses. Moderators remove rule violations, not opinions.

Technical integration Bot integrations (e.g., Discord/GitHub webhooks) alerting faculty to trending community issues, feeding directly into syllabus backlog updates.
Data & KPIs available
  • Channel message volume
  • #suggest-a-topic upvotes
  • Member growth rate
  • Bot-flagged trending issues

Ties an individual Discord member to a CityU learner record only once accounts are linked — see the Identity & tracking layer breakdown in Next Steps.

Ebook / Gated Content

Lead-Generation Downloads
Strategy
Trade a free, high-value asset (program guide, partnership overview, research brief) for an identified email — the one touchpoint in this ecosystem where the visitor volunteers their own identity, no account-linking required.
  • Hosting

    GitHub Pages, same as the rest of the ecosystem's static assets — but GitHub Pages is static-only, so form submission needs an external handler (no built-in backend/database).

  • Form mechanism — recommended

    Custom-branded HTML form on the page, submitted silently (via fetch(), no-cors) to a Google Form's /formResponse endpoint using its entry.XXXXX field IDs. Free, unlimited submissions, responses land in a Google Sheet in real time, and the visitor never sees Google's UI — matches the site's own design instead.

  • Simpler fallback

    A third-party form endpoint (Formspree, Web3Forms, FormSubmit) if the team wants zero JavaScript — same custom-HTML styling, but with a monthly submission cap on most free tiers (FormSubmit is the one unlimited option).

  • Fields — kept to 3–4, on purpose

    Full Name, Email (required, validated), "I am a…" dropdown (Prospective Student, Current Student/Alumni, Corporate/Employer Partner, Educational/Community Partner, Other), Organization/Company (optional). Every extra field costs 5–10% conversion on gated content — this is not the place for a full RFI form.

  • Reusable across every ebook

    Generic title ("Request Resource Access") and description, plus one "Resource Requested" dropdown field — one form serves every future ebook/whitepaper instead of building a new one each time. A pre-filled link per landing page can default that field automatically.

  • Delivery

    Immediate on-screen download link in the confirmation message — matches the friction-reduction principle (don't make them check email for something they expect instantly). Email delivery works as a backup/re-engagement touchpoint, not the primary path.

Technical integration Hidden UTM fields (utm_source, utm_campaign) on the form, carried through to the Google Sheet / CRM — ties every download back to the touchpoint that drove it (a specific YouTube video, seminar, workshop), feeding the same Identity & tracking layer as the rest of the funnel.
Data & KPIs available
  • Identified email (not aggregate)
  • Persona segment
  • Organization / partner type
  • Download count per resource
  • UTM source (if instrumented)

The strongest identity signal of any touchpoint in this ecosystem — unlike YouTube, the visitor volunteers a real, usable email address directly into the funnel.

Alternative Path — Practitioners Without Open edX (Yet)

Real example: STRIDE for AI Agents
Real example
A practitioner-publisher running the funnel entirely outside Open edX: a YouTube talk ("STRIDE Analysis on Agentic Systems") pointing to a free, self-hosted 6-module course on GitHub Pages — STRIDE for AI Agents — instead of an Open edX micro-course.
TOFU · YouTube talk → MOFU · GitHub Pages course (no Open edX) → Conversion · CityU Programs + Apply

BOFU is skipped entirely — no Community Hub, Workshop, or Seminar touchpoint in the video description or pinned comment.

Why this still counts This matches the Ecosystem Playbook's framework-vs-platform distinction — a practitioner can run this whole pattern with zero shared infrastructure: no Open edX account, no CityUX Discord dependency, nothing to provision. It's proof the framework travels beyond STC's own tooling, not a workaround.
Gap
No BOFU touchpoint means no community capture and no tracking — there's currently no way to tell how many viewers of this video actually convert (ties to the SSO & tracking layer in Next Steps).

1

Top of Funnel — Discovery & Awareness

User state
Viewer lands on a YouTube video lecture or concept explainer (via search, algorithm recommendation, or shared link).
  • Video content

    High-value, modular technical explainer addressing a specific concept or pain point.

  • In-video on-ramps

    On-screen QR codes, pinned comments, and description links prompting viewers to access the underlying hands-on assets.

  • Interactive end screens & cards

    Clickable cards driving traffic directly to a corresponding Open edX pilot module or GitHub starter code.

2

Middle of Funnel — Engagement & Skill Building

User state
Learner moves from passive viewer to active practitioner by engaging with open resources.
  • GitHub starter repositories

    Click-to-launch GitHub Codespaces links (found in the YouTube description) allow learners to immediately run and test the lab code without local environment setup.

  • Open edX micro-courses

    Free self-paced pilots hosting embedded YouTube lectures alongside auto-graded quizzes and interactive code blocks.

  • Automated feedback & badging

    GitHub Actions run test suites against learner commits, while Open edX tracks unit completions and issues digital completion badges.

3

Bottom of Funnel — Consideration & Cohort Validation

User state
Highly engaged learner seeking structured feedback, community interaction, or live instruction.
  • Community Hub invitations

    Automated email or platform nudges inviting badge earners/repo forks into Discord, Slack, or GitHub Discussions to interact with peers and TAs.

  • Live workshops & seminars

    Targeted invites to attend live, cohort-based hands-on workshops or guest seminars for deeper topic validation.

  • Automated nurture sequences

    Email workflows triggered by Open edX course completion, highlighting credit transfer options, certificate pathways, and micro-credentials.

4

Conversion — Formal Academic Enrollment

User state
Learner transitions from open ecosystem participant to matriculated City University of Seattle student.
  • Credit articulation

    Micro-credentials and Open edX pilot achievements map directly into prerequisite credits for degree programs.

  • Direct course & program registration

    Dedicated landing pages streamlining application and registration into official CityU syllabi, courses, and programs.

  • Faculty & advisory touchpoint

    TAs and faculty active in the Community Hub provide direct academic counseling and onboarding.

User Stories & Journeys

The funnel isn't one path — four learners, four entry points, four different "wins"
🎥
The Self-Directed Learner
Organic discovery via search or algorithm recommendation
TOFU · YouTube (official or practitioner-published) → MOFU · GitHub + Open edX → BOFU · Community Hub + Workshop → Conversion · Program registration

The "default" path this ecosystem is built around — see Learner Conversion Funnel for the full stage-by-stage breakdown. Most predictable journey to design touchpoints and nurture sequences around. Covers content from an official CityU channel or an individual practitioner-publisher, as long as that content routes into Open edX — see the next card for what happens when it doesn't.

🎙️
Discovered via Industry Practitioner Content
Finds CityUX indirectly — through a practitioner's own YouTube/blog/GitHub content, not an official CityU channel
TOFU · Practitioner-published content

A different axis from the other cards here: this describes who published the content (an individual practitioner building trust under their own name — see the Ecosystem Playbook's practitioner-publisher model), not a new destination. What happens next depends entirely on whether that practitioner has adopted Open edX:

If Open edX-integrated

Collapses into The Self-Directed Learner journey above — same MOFU/BOFU/Conversion path.

If not yet on Open edX

Follows the compressed Alternative Path — GitHub Pages course, BOFU skipped, straight to Conversion. This is the STRIDE for AI Agents case, today.

🚔
The Frontline Professional
e.g. an SPD officer or chief, reached through an institutional partnership
BOFU · Workshop (skill-up cohort) → MOFU · Open edX compliance modules → Conversion · Department-recognized credential

Skips TOFU discovery entirely — the department partnership is the entry point. "Conversion" here means a workplace-recognized micro-credential or compliance certificate, not CityU degree enrollment. Ties directly to the Law Enforcement / Public Sector AI Program.

💬
The Community-First Explorer
Finds CityUX through word of mouth, a starred GitHub repo, or a Discord invite from a peer
BOFU · Community Hub → MOFU · Open edX + GitHub (backfilled) → Conversion · Micro-credential or program interest

Enters "out of order" — proof the funnel isn't strictly linear. Every touchpoint needs to work as an entry point on its own, not just as a handoff from the stage before it.

🌏
The Prospective International Student
Reached via direct seminar outreach or a cultural-exchange invitation
TOFU · Seminar / direct outreach → MOFU · Open edX pilot → BOFU · Community Hub (peer connection) → Conversion · Formal international enrollment

Highest-value outcome — full degree enrollment, not a micro-credential — but the longest cycle. Needs a dedicated faculty/advisory touchpoint out of the Community Hub, not just automated nurture emails.

Why this matters "Conversion" isn't one metric. A department partner converts into a compliance credential, a community-first learner converts into program interest, an international student converts into full enrollment. The analytics dashboard (Part 6) needs to track these as separate outcomes, not one blended conversion rate.

Next Steps for Deployment

Three things to build before the loop can run end-to-end
  1. 1
    Identity & tracking layer
    Open edX as the profile hub for what can actually be linked (Open edX, GitHub, CRM); UTM-only referral tracking for YouTube. See feasibility breakdown below — this is not one uniform build.
  2. 2
    Template pipeline
    Standardize the GitHub repository template structure (including README.md layouts, Codespace devcontainers, and feedback forms) across all workshops and labs.
  3. 3
    Analytics dashboard
    Aggregate completion analytics from Open edX, engagement metrics from YouTube, and commit activity from GitHub to quantify conversion funnel health for institutional stakeholders.
Feasibility breakdown — identity & tracking layer
"SSO & tracking layer" is really three different problems wearing one label — worth splitting apart before scoping the work:
Open edX ↔ GitHub ↔ CRM

Low-medium effort. Open edX already runs Django OAuth2 provider support — the same mechanism edX itself uses for its mobile app and internal microservices. Register the CRM/dashboard as OAuth2 clients and pull profile + completion data from Open edX's REST API. Mostly configuration, not new infrastructure.

GitHub account linking

Medium effort (~1–2 weeks). A learner's personal GitHub account can't be SSO'd through Open edX — GitHub doesn't support swapping identity providers for individual accounts. Instead, build a "Connect your GitHub" OAuth linking flow in the learner's Open edX profile using the existing third_party_auth extension point, and store their GitHub username against their Open edX record.

YouTube → downstream

Hard ceiling, not a difficulty problem. YouTube never exposes individual viewer identity — by design, not something you can engineer around. The only trackable signal is a UTM-tagged link click (referral-level: "12% of signups came from Video X"), never identity-level ("this specific viewer enrolled").

Bottom line Use Open edX as the identity/profile hub for everything that can actually be linked — Open edX, GitHub (via account-linking), and the CRM. Don't scope YouTube into "unified learner profile": track it as referral/click attribution only, via UTM parameters, and treat that as a separate, permanently aggregate-only metric rather than a gap to eventually close.

Prioritization — Effort vs. Impact

Everything from the sections above, sorted into what to build first

Columns: effort (low → high, left to right) · Rows: impact (high → low, top to bottom)

Quick Wins
Low Effort · High Impact
  • UTM parameters on all outbound links (YouTube, practitioner content) — no infrastructure needed, immediate referral attribution
  • Open edX certificate copy mentioning CityUX — configuration only (done)
  • Open edX OAuth2 client registration for CRM/dashboard — config-only, unlocks the profile-hub strategy
  • Gated ebook download form (branded HTML → Google Form silent submission) — a few hours of setup, and the strongest identity signal in the whole ecosystem
Major Projects
High Effort · High Impact
  • Analytics dashboard — aggregates Open edX, YouTube, and GitHub data into one funnel-health view for stakeholders
  • GitHub account-linking OAuth flow (~1–2 weeks) — bridges the two richest data sources
  • Template pipeline standardization across all workshops/labs — needs faculty coordination, not just dev time
  • Automated certificate → nurture email sequence — directly drives Conversion
Fill-Ins
Low Effort · Low Impact
  • YouTube Data API metadata sync to Open edX units — convenience automation, doesn't move conversion
  • Seminar automated post-event content packaging — nice-to-have, not funnel-critical
  • Community Hub bot integrations (Discord/GitHub webhooks → faculty alerts) — low lift, low reach today
Reconsider
High Effort · Low Impact
  • Full unified cross-platform SSO including YouTube — the identity ceiling is by platform design, not effort. No amount of engineering closes it; money and time here buys little beyond what UTM + OAuth linking already deliver.
