Section 02

Module Deep-Dive: AI-Augmented Governance

PE03 demonstrates how to separate the agent that modifies code from the agent that validates it—a foundational principle for autonomous quality assurance at scale.

🏛
Core Principle: Separation of Concerns Between Agents

The orchestrator agent repairs a broken production service while the evaluator agent independently audits the result against ITIL Incident Management standards. Neither agent can see the other's internal state.

# PE03 Two-Agent SRE Governance Pipeline incident_router.py (immutable baseline — never touched) │ ▼ agent_debugger.py (Orchestrator — triage & repair) │ writes ▼ incident_router_healed.py ──► incidents.json ──► triage_summary.md ──► debugging_report.md │ ▼ eval_debugger.py (Evaluator / Governance Gate — independent audit) │ writes ▼ eval_results.json { "status": "APPROVED"|"REJECTED", "score": 100|0, ... }

⚙️ Orchestrator Agent

  • Reads incident_router.py but never mutates it
  • Intercepts NotImplementedError from runtime stderr
  • Runs up to 5 convergence iterations
  • Generates incidents.json with ITIL fields: severity, category, summary, remediation
  • Produces triage_summary.md and debugging_report.md

🔍 Evaluator / Governance Gate

  • Runs completely decoupled from the orchestrator
  • Validates healed artifacts against ITIL Incident Management rubric
  • Emits binary decision: APPROVED or REJECTED
  • Score field enables partial credit tracking over iterations
  • Prevents self-validation bias—no shared memory with orchestrator
Section 03

Module Deep-Dive: Spec-Driven Development with AI

HOS10A shows students how to derive every architecture decision—routes, streaming design, database indexes, and prompt construction—from a single client conversation before touching the keyboard.

# HOS10A RAG Stack (all decisions derived from client conversation) React (Frontend, port 3000) │ POST /chat (SSE stream) ▼ Node/Express (Backend, port 5050) │ $text search (limit 5) POST to Ollama API ▼ ▼ MongoDB Atlas (records collection) → Gemma 2:2b (Ollama, port 11434) text index on: name, position # RAG Flow: User query → MongoDB $text search → top 5 results as context → augmented prompt → Gemma 2:2b → SSE stream → React UI
1

Client Conversation as Source of Truth

A Slack-style thread between Product Manager (Maria Chen), Developer (Alex Rivera), and Tech Lead (Sam Park) embeds all spec decisions. Students read the conversation before writing any code.

2

Spec Extraction

Students parse the conversation to extract API route names, SSE headers, MongoDB query strategy, prompt construction pattern, and streaming terminator—creating a formal spec document.

3

AI-Assisted Implementation

With the spec in hand, students use Claude or Cursor to scaffold each layer, validating that generated code matches the spec before moving to the next layer.

4

Spec Compliance Verification

Final deliverable maps every implementation decision back to a specific line in the client conversation—demonstrating traceability from requirement to code.

Persona Role Spec Contribution
Maria Chen Product Manager Defines feature goal, stack choice, and RAG flow requirements
Alex Rivera Developer Asks clarifying questions (e.g., SSE vs. regular API calls, error handling)
Sam Park Tech Lead Specifies exact implementation: route names, SSE headers, MongoDB query, prompt construction, streaming terminator
Section 04

Module Deep-Dive: AI Assisted Documentation

Students use generative AI to produce high-quality, architecturally grounded README summaries for every lab—eliminating vague boilerplate and reducing cognitive load through specific, pre-scoped prompts.

💡
Design Principle: Specific Prompts Over Generic Templates

A generic template increases cognitive load and produces poor AI output. Each lab ships with exact values for [Lab Title], [Feature], and [Tool] so students get architecture-quality summaries on the first try.

1

Select a Key Code Snippet

Student identifies the most architecturally significant code they modified in the lab.

2

Structured AI Prompt

"I am a student completing a technical lab on [Lab Title]. I have implemented [Feature] using [Tool]. Based on this code snippet… Please write a concise, professional 3-sentence summary for my GitHub README that explains the architectural role of this code and how it contributes to a 'Smart and Secure System.'"

3

Review, Edit, and Commit

Student validates the AI output for accuracy, edits as needed, and pastes the result into README.md.

Section 05

Pedagogy Modules: Predictive Learning & Socratic AI Tutor

Two modules target the learning process itself—one primes students before the lab, the other replaces passive answer-giving with guided discovery during the lab.

Module 05 — Predictive Learning
1

Phase 1 · AI Prompting (5 min)

Before opening TryHackMe, the student asks the AI: "What are the three most critical concepts I should look for in a lab on [topic]? Give me a cheat sheet."

2

Phase 2 · Active Lab (15–20 min)

Student completes the assigned tasks while actively hunting for the AI's three predicted concepts.

3

Phase 3 · Integrated Summary (5–10 min)

150–200 word reflection: What did you learn? Did the AI predict correctly? What pro tip did you actually see in action?

Module 06 — Socratic AI Tutor

🧠 The "Guide on the Side" Pattern

  • Probe: Student presents a problem
  • Check: AI asks what they've already tried
  • Scaffold: AI offers a smaller bridging question
  • Discovery: Student reaches the answer; AI confirms and advances

🏆 Notable Implementations

  • Khanmigo — explicitly tells students "I won't give you the answer"
  • Code.org AI Tutor — asks students to explain what a line of code does before debugging
  • Socratic by Google — analyzes the question to guide through prerequisite concepts

Section 07

Curriculum Rollout Plan

A phased approach integrates AI infusion progressively across the STC program—starting with foundational documentation habits and advancing to autonomous multi-agent governance.

Phase 1 · Foundational
AI as a Writing Tool
  • Deploy AI Assisted Documentation across all TT labs
  • Introduce Pre-Flight Briefing for security/network labs
  • Students interact with AI as a drafting assistant
Phase 2 · Applied
AI as a Co-Pilot
  • Augmented Project Planning integrated in PM track
  • Cursor vs Claude comparison introduced in full-stack courses
  • Socratic Tutor deployed for office hours and self-study
Phase 3 · Advanced
AI as an Autonomous Agent
  • Spec-Driven Development as capstone workflow
  • AI-Augmented Governance (PE03 multi-agent pipeline)
  • Students build and evaluate their own governance agents
